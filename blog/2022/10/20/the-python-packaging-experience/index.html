What should the "default" Python packaging experience look like?

October 20, 2022 · 7 minutes

Why I’m writing this#

This has been a topic that’s been on my mind since ~2019 and it has come up repeatedly in discussions around Python packaging but… it’s always mentioned as a “we should think about this” or “here’s one way we could do things” with… well, not much happening after other than a discussion (mostly because the actual work here is unbounded in theory).

Somewhat recently, Paul Moore said:

At some point, I do think that pip needs to make a firm decision on whether it’s a development workflow tool or just an installer

Somewhat earlier, Donald Stufft said:

All that being said, I think trying to follow the “unix philosophy” is a mistake and is actually a pretty poor UX. Yea a lot of nerds grok it because we’ve caused enough collective brain damage by being forced to use it over time and it works better for the typical unix tools because they generally just come preinstalled. I think it would just add additional complexity to an already confusing landscape of tools for our end users.

The pip is not the same as python -m pip problem#

The python problem#

Python doesn’t have a good interpreter version management story. There is no “single” way to say “Get me Python 3.9 and Python 3.10” across supported platforms in a consistent manner.

“Too complicated”#

Let’s look at the “out of the box experience” for authoring and publishing a package; using what came with Python and the “dedicated” tools for that task.

To get started, you’ll need to create a virtual environment with venv (python -m venv).

Oh, and don’t forget to pip install wheel inside it, or some packages will use setup.py install and have a degraded install3. And, make sure you’re using the right Python from PATH. Oh, and make sure pip and python point to the same thing. Oh, and you might want to use python3.x instead of python or python3 because those could point to something your OS provided that you really shouldn’t use – except, that it’ll seem like it works.

All these are subtle failure modes. They exist because – your python may not be the Python you want and may be different from the pip you have.4.

We can embrace that our default out of the box experience will be a largely incoherent one – where the process of authoring to publishing a package will involve, at minimum, 5 different tools (pip, venv, a build-backend, build, twine). In that view, pip is merely an installer. That’s meaningfully diffferent from the current state of thing: it gives us a clear answer to that, and a clear focus long-term – there’s many “bugs”, feature requests and content-for-documentation that can be trimmed given this much smaller scope for the tool – a package installer.

Roadblocks to improving things#

Python Lockfiles: A case study#

Among the seven folks who have the label “pip maintainer”, I can safely say that we share to have a general sense that because pip is a tool that ships with Python, and this is a privedged position within the Python Packaging ecosystem. Implementing certain bits of new functionality that don’t go through a concensus building process that involves every-other-tool in the ecosystem, is a detriment to everyone involved.

To pick an example, let’s take lockfiles: it’s technically feasible to implement an arbitrary lockfile format in pip, that evolves with pip, in lock-step with it (we see that’s implemented already in the form of pip-tools). Given the priviledged position that pip has within the ecosystem (i.e. ships with the language), whatever it does would become the “de-facto” standard and commerical tools will add support for that format much quicker (eg: requirements.txt) than something similar from PDM, Poetry or Pipenv.

As of today, this means that for implementing something “important” in pip, we’re effectively required to going through a somewhat-exhaustingly long process of a non-iterative, waterfall-style design process for a lockfile format (PEP 665’s slightly iterative process of “we’ll cover the more complicated case later is part of what caused its demise). This is also while trying to satisfy needs to tools that don’t use the same resolution logic – which, we’ll see how that goes, but it’s definitely got a much longer feedback loop associated with it.

In summary, we’re in a state where the process for adding new functionality to our default “out of the box” experience is designed to be both very difficult and very “energy” intensive – leading to it being pretty slow. The effect of this is that our “out of the box” experience is always going to be worse than the experience with other tools in the ecosystem.

Is that what we want?

The competition model for Python build-backends#

Python packaging now has a competition model for build-backends – i.e. tools that handle the generation of installable artifacts from source trees.

Reading “The community spectrum: caring to combative” - Insight From Alex Bayley, the model flagged something amusing to me.

The Competitive Spectrum describes communities as being:

  • Caring: members are motivated by helping each other.
  • Collaborative: members share goals and help each other to achieve them.
  • Cordial: members have their own goals which do not conflict with each other.
  • Competitive: members share the same goals, and compete against each other to achieve them.
  • Combative: members must achieve their goals by preventing others from being doing so.

With the decoupling of the build-backends to allow alternatives to setuptools/distutils to be built, we’ve also moved from a collaborative (or at least, cordial) model for the build mechanisms, to a competitive model for that space.
